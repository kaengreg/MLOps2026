# NYC Yellow Taxi Trip Price Prediction MVP
MVP-пайплайн для задачи предсказания стоимости поездки (`total_amount`) на датасете **NYC Yellow Taxi Trip Data**.

## Цель проекта
Предсказывать `total_amount` по характеристикам поездки.

В качестве признаков используются только те поля, которые доступны без раскрытия целевой переменной. Компоненты итоговой цены и сам `total_amount` исключаются из обучения, т.к. иначе модель может научиться выстраивать `total_amount` по ним.

## Как запускать этапы

Ниже приведен рекомендуемый порядок запуска этапов.

```bash
chmod +x data_loading.sh && ./data_loading.sh 
python run.py --mode prepare_data
python run.py --mode train
python run.py --mode inference --file ./path_to_file.csv
python run.py --mode update
python run.py --mode summary
```
#### Запуск через GitHub Actions

Workflow-файлы расположены в директории .github/workflows/.

Для CI/CD-сценариев используется следующий подход:

* basic-scenario.yml — единичный независимый запуск обучения;
* iterative-scenario.yml — начальное обучение + несколько шагов update;

### 1. Подготовка данных

Команда:
```bash
python run.py --mode prepare_data
```

Формирует временные батчи из исходного датасета и сохраняет их в data/raw_batches/.

### 2. Обучение моделей

Команда:
```bash
python run.py --mode train
```

Обучает несколько моделей, выбирает лучшую и сохраняет артефакты обучения.

### 3. Инференс на внешних данных

Команда:
```bash
python run.py --mode inference --file ./path_to_file.csv
```
Применяет сохраненную лучшую модель к новому CSV-файлу и сохраняет предсказания.

### 4. Обновление моделей на новом батче

Команда:
```bash
python run.py --mode update
```

Обрабатывает следующий необработанный батч и обновляет модели.

### 5. Построение итогового отчета

Команда:
```bash
python run.py --mode summary
```
Формирует итоговый markdown-отчет по логам качества данных, обучения и update.

Подробное описание логики каждого режима приведено в [`doc/task.md`](./doc/task.md).

## CI/CD сценарии
### Сценарий 1. Базовый независимый запуск
Workflow автоматически запускается при `push` и `pull_request`, поднимает Python-окружение, устанавливает зависимости и выполняет один независимый запуск обучения модели.

Особенности:
- используется ограниченное число батчей для быстрого воспроизводимого CI-запуска;
- основной артефакт — текстовый файл лога;
- дополнительно сохраняются training log и сериализованная лучшая модель.

### Сценарий 2. Итеративное независимое обучение
Workflow выполняет:
1. начальное обучение модели;
2. несколько последовательных шагов `update`.

Количество итераций задается в YAML-файле workflow. Перед запуском сценария состояние пайплайна и старые артефакты очищаются, чтобы каждый CI-запуск был независимым и воспроизводимым.

Артефакты:
- текстовый лог сценария
- `training_log.csv`
- `update_log.csv`
- лучшая сериализованная модель
- `pipeline_state.json`

## Аргументация выбранного подхода

В проекте выбран подход поэтапного расширения MVP.

На первом этапе реализован локальный MLOps-пайплайн для потоковой обработки табличных данных:
- подготовка временных батчей
- контроль качества данных
- обучение нескольких моделей
- выбор лучшей модели
- инкрементальное обновление на новых батчах
- сохранение артефактов и итогового отчета

На втором этапе поверх этой логики добавляются GitHub Actions workflow. Такой подход выбран по следующим причинам:

1. **Повторное использование уже реализованного пайплайна**  
   Workflow не дублируют ML-логику, а вызывают уже существующие сценарии `train`, `update`, `summary`.

2. **Разделение сценариев по уровню сложности**  
   Базовый сценарий позволяет быстро проверить корректность окружения и обучения. Итеративный сценарий расширяет эту схему до последовательного update. 

3. **Воспроизводимость**  
   Для CI-запусков используются заранее подготовленные батчи из репозитория, а также фиксированные зависимости в `requirements.txt`.

4. **Контроль состояний**  
   Все ключевые этапы сохраняют артефакты: логи обучения, логи update, сериализованные модели, метаданные моделей и состояние пайплайна.
