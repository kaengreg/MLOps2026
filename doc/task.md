# Описание задачи

## 1. Постановка задачи
Цель проекта — построение MLOps-пайплайна для потоковой обработки табличных данных и предсказания `total_amount` на данных поездок NYC Yellow Taxi.

Проект реализует минимально рабочую ML-систему, которая:
- принимает табличные данные
- разбивает их на временные батчи
- контролирует качество данных
- обучает несколько моделей
- выбирает лучшую модель
- поддерживает инференс на новых данных
- поддерживает последовательное обновление модели на новых батчах
- сохраняет артефакты обучения и мониторинга

## 2. Источник данных
Используется датасет **NYC Yellow Taxi Trip Data**.

В проекте используются две части данных:
- `yellow_tripdata_2015-01.csv` — основная обучающая часть; из нее формируются временные батчи, сохраняемые в `data/raw_batches/`
- `yellow_tripdata_2016-01.csv` — внешняя часть; используется для проверки inference-сценариев и может быть сохранена в `data/external/`



Использование данных:
- основной обучающий поток формируется из батчей, подготовленных из `yellow_tripdata_2015-01.csv`
- внешние данные для проверки и inference формируются из `yellow_tripdata_2016-01.csv` или из аналогичных внешних CSV-файлов

## 3. Целевая переменная
Целевая переменная: `total_amount`.

Не используются признаки, напрямую раскрывающие целевую переменную через компоненты итоговой стоимости. Это сделано для того, чтобы модель не восстанавливала `total_amount` напрямую по составным частям цены.

## 4. Организация потоковых данных
Исходный датасет сортируется по времени `tpep_pickup_datetime` и разбивается на последовательные временные батчи.

Батчи сохраняются в файловой системе и используются для:
- начального обучения моделей
- последовательного обновления моделей
- инференса
- формирования итоговых отчетов

Для обучения и update используется директория `data/raw_batches/`
Для внешних проверок и inference используется директория `data/external/`

## 5. Data Quality
Для каждого батча рассчитываются и сохраняются следующие метрики качества данных:
- число строк
- число пропусков
- число дублей
- количество отрицательных значений целевой переменной
- количество отрицательных значений `trip_distance`
- количество невалидных координат
- количество некорректных длительностей поездки
- верхние квантили по ключевым числовым признакам

Метрики качества сохраняются в `dev/metrics/data_quality_log.csv` и затем используются в summary-отчете.

## 6. Очистка и подготовка данных
На этапе подготовки данных выполняются
- удаление пропусков в ключевых полях
- фильтрация отрицательных значений
- фильтрация выбросов по квантилям
- фильтрация некорректных координат
- удаление дублей

Генерируемые признаки:
- `pickup_hour`
- `pickup_weekday`
- `pickup_month`
- `trip_duration_min`

Также выполняется разделение признаков на:
- категориальные
- числовые

Для разных моделей используются соответствующие варианты предобработки:
- заполнение пропусков
- one-hot encoding для категориальных признаков
- масштабирование числовых признаков для нейросети
- нативная обработка категориальных признаков для CatBoost

## 7. Модели
В проекте рассматриваются следующие модели:
- `CatBoostRegressor`
- `DecisionTreeRegressor`
- `MLPRegressor`

Модели обучаются на одном и том же наборе признаков и сравниваются по качеству на test-батче. По итогам обучения выбирается лучшая модель.

## 8. Гиперпараметры
### CatBoostRegressor
Перебираются:
- `iterations`
- `depth`
- `learning_rate`

### DecisionTreeRegressor
Перебираются:
- `max_depth`
- `min_samples_split`
- `min_samples_leaf`

### MLPRegressor
Перебираются:
- `hidden_layer_sizes`
- `alpha`
- `learning_rate_init`

Подбор гиперпараметров выполняется как небольшой Grid Search на валидационной части обучающего набора.

## 9. Валидация
Используется схема hold-out:
- `train` строится на предыдущих батчах
- `test` — последний доступный батч
- для подбора гиперпараметров внутри train дополнительно выделяется validation split

Для каждой модели рассчитываются:
- `MAE`
- `RMSE`
- `R2`

Метрики обучения сохраняются в `dev/metrics/training_log.csv`.

## 10. Сохранение артефактов
В проекте сохраняются:
- логи качества данных
- логи обучения
- логи update
- versioned/current/best модели
- метаданные моделей
- итоговый отчет
- pipeline state

Основные артефакты:
- `dev/metrics/data_quality_log.csv`
- `dev/metrics/training_log.csv`

## 11. Сценарии запуска
Проект поддерживает следующие режимы запуска через `run.py`:

### `prepare_data`
Подготавливает поток батчей из исходного датасета и сохраняет их в файловую систему.

### `train`
Обучает несколько моделей, выполняет небольшой подбор гиперпараметров, сравнивает модели и сохраняет лучшую.

### `inference`
Загружает сохраненную лучшую модель и строит предсказания на новом CSV-файле.

### `update`
Берет следующий необработанный батч из потока и обновляет модели. Для разных моделей используется либо инкрементальное обновление, либо переобучение на накопленных данных.

### `summary`
Формирует итоговый markdown-отчет на основе логов качества данных, обучения, update и текущих метаданных лучшей модели.

## 12. CI/CD сценарии
Во втором этапе проект расширяется workflow-сценариями на GitHub Actions.

### Базовый workflow
Один независимый запуск обучения модели в CI.  
Workflow автоматически запускается при `push` и `pull_request`, поднимает окружение, запускает обучение и сохраняет текстовый лог как artifact.

### Iterative workflow
Один CI-запуск, который выполняет:
1. начальное обучение
2. несколько последовательных шагов `update`

Количество update-итераций задается в YAML-файле workflow.
